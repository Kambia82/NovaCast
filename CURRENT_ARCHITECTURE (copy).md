| Layer | Technology | Notes |
|---|---|---|
| Framework | React 18 | JSX/TSX mix (see §5) |

| Persistence | `localStorage` | Tacklebox, memories, wishlist — two conflicting storage keys (see §5) |
| PWA | Service worker + `manifest.json` | Registered in `main.tsx`; icon assets in `public/` |

## 3. Is the App Using Firebase, Supabase, or Both?

**Supabase for everything data-related. Firebase for hosting only.**

| Service | Role | Evidence |
|---|---|---|
| **Supabase** | PostgreSQL database (3 tables), Row Level Security, anonymous/authenticated access | `src/lib/supabase.ts`, `supabase/migrations/` |
| **Firebase** | Hosting deployment target — serves `dist/` as a SPA | `firebase.json` (site: `novacast-26e4c`) |
| Firebase SDK | **Not installed, not imported, not used** | Not in `package.json` `dependencies` or `devDependencies` |

The Firebase JavaScript SDK is completely absent from the application. Firebase is purely a CDN/hosting mechanism from the Bolt export. The Replit migration does not need Firebase at all.

## 4. Screens, Routes, and Navigation Flows

Novacast uses **no router**. Navigation is entirely driven by React state variables in `App.tsx`.

### State variables controlling view
```typescript
const [view, setView] = useState<AppView>('welcome');
// type AppView = 'welcome' | 'wizard' | 'result' | 'tacklebox' | 'reference'

const [screen, setScreen] = useState(0);       // wizard sub-step (0–5)
const [showReference, setShowReference] = useState(false); // modal overlay
const [showAdmin, setShowAdmin] = useState(false);         // admin panel (hash-triggered)
```

### Key observation on the Welcome screen GPS flow
The welcome screen has its **own** GPS implementation using the Overpass API that searches for *any* water body on OpenStreetMap within 30 km. This is **completely different** from the Wizard's GPS flow, which only searches the app's own Supabase database of known spots. The welcome screen Overpass flow was later superseded by the Wizard's in-database GPS flow — the two do different things and are essentially parallel, competing implementations.

## 5. Unused, Duplicate, Legacy, or Abandoned Code

### Duplicated constants (same data in two files)

| Constant | App.tsx | NovaCastWizard.jsx | Impact |
|---|---|---|---|
| `RECENT_WEATHER_OPTIONS` | ✓ (6 items) | ✓ (6 items, slightly different desc text) | Two slightly different descriptions for the same options |
| `REGION_LABELS` | ✓ (9 entries) | ✓ (8 entries — missing `'custom'`) | Wizard is missing the `custom` region |

### Conflicting localStorage keys

| Key | Set by | Reads | Contents |
|---|---|---|---|
| `novacast_tacklebox` | `App.tsx` useEffect | `App.tsx` initial state | `{ lures, colors, walmart }` |
| `novacast_tacklebox_v2` | `NovaCastTacklebox.tsx` | `NovaCastTacklebox.tsx` | `{ lures, colors, walmart, memories, wishlist }` |

The tacklebox component (v2 key) was upgraded to add memories and wishlist, but `App.tsx` still seeds the tacklebox state from the **old key** (`novacast_tacklebox`). A user who saved items before the upgrade will have their lures/colors in the old key but memories/wishlist in the new key. They're partially merged on mount in `NovaCastTacklebox.tsx` via `{ ...loadStorage(), ...externalTacklebox }` but this is fragile.

### Plain JavaScript file in a TypeScript project
`NovaCastWizard.jsx` is a `.jsx` file with no TypeScript annotations. All props receive `any` types. This is the most complex and longest component in the app and has no type safety.

### Hardcoded secrets
- **OpenWeatherMap API key** hardcoded in `NovaCastWizard.jsx` line 218: `appid=9e751a40a370416832496e123e1098cc`
- **Admin password** hardcoded in `App.tsx` line 184: `'castmaster2025'`

### Auth schema with no auth UI
The Supabase migration creates `custom_lakes.user_id` referencing `auth.users`. The RLS policies require authentication to read/write custom lakes. However, the app has **no login or signup UI** and no Supabase Auth flow. The `supabase.auth` module is never called anywhere. This means:
- `custom_lakes` can never be successfully written (the insert will fail without an authenticated session)
- The custom lake form in the admin panel likely silently fails

### The `novacast_source.html` file
An older single-file HTML prototype with a completely different color palette (green `#4fc97e` accent instead of the current teal `#00e5c7`). This is dead weight — it predates the current React app and is not referenced anywhere.

### The `FULL_PROJECT_COPY.txt` file
A dump of all source code as a text file. Not used at runtime. Legacy export artifact.

### Wizard `calcDistance` duplicated
The haversine distance formula is implemented independently in both `App.tsx` and `NovaCastWizard.jsx` (as `calcDistance`). Same formula, different function names, no shared utility.

### Welcome screen Overpass API GPS flow
As noted in §4, the welcome screen's "Find Near Me" uses the Overpass API to find *any* OpenStreetMap water body. This was probably the original discovery mechanism before the Supabase database was populated. The Wizard's location screen now provides a better, curated GPS experience. The Overpass flow adds complexity and an external API call for a result that is less useful than the Wizard's own GPS flow.

## 6. How Maps, Locations, Fishing Spots, Species, Tackle, and User Profiles Currently Work

### Maps / Locations
- **No embedded map UI.** Coordinates are used only to compute distances and generate Google Maps deep-links.
- Google Maps links: satellite view link (`/maps/@lat,lon,15z/data=...`) and directions link (`/maps/dir/?api=1&destination=lat,lon`)
- Nominatim used for: zip-code geocoding (App.tsx welcome), city name geocoding (Wizard screen 5)
- Overpass API used for: finding raw OSM water bodies near GPS position (App.tsx welcome only)

### Fishing Spots
Three sources merged at runtime:

| Source | Storage | Who adds | RLS |
|---|---|---|---|
| `water_bodies` | Supabase | Developer (migration data) | Any authenticated user reads |
| `admin_lakes` | Supabase | Admin via `#admin` password panel | Any authenticated user reads; service role writes |
| `custom_lakes` | Supabase | User via personal form | User sees own only (but broken — no auth) |

Within each spot (`water_bodies.spots` JSONB array), each entry has:
```json
{ "name": "string", "detail": "string", "shallow": bool, "deep": bool, "always": bool }
```
The `getSpots()` function in `recommendations.ts` re-sorts these spots based on time-of-day and sky conditions.

### Species
Species are stored as a `text[]` array on each water body row. They are display-only — used to show tags on water body cards in the wizard. Species does not affect the recommendation engine; the user's fish target (selected in wizard screen 0) drives lure/color recommendations instead.

### Tackle / Recommendations
The entire recommendation engine lives in `src/data/recommendations.ts` as pure TypeScript functions — no database queries. Inputs from the wizard flow through:

```
getLures(sky, water, temp, fish, time, pressure?)  → Lure[] (scored + sorted)
getColors(sky, water, time)                         → ColorResult
getWalmart(fish, water, time)                       → WalmartItem[]
getProTip(sky, water, temp, wind, fish, time, ...)  → string
getFishMovement(time, sky)                          → FishMovement { title, depthPct, moveText }
getBarometricImpact(pressure)                       → { title, text, depthAdj, activityMod }
getRecentWeatherImpact(recentWeather[])             → string[]
```

### User Profiles
**There are no user profiles.** User identity is entirely absent from the UX. The only user-specific persistence is `localStorage` (tacklebox, memories, wishlist). The Supabase `auth.users` table is referenced in the migration schema but the auth flow was never built.
